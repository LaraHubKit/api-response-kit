# LaraHub API Response Kit – v1 Technical Specification

## 1. Overview

**Package Name:** LaraHub API Response Kit
**Vendor:** LaraHub
**Domain:** [https://www.larahub.io](https://www.larahub.io)
**Repository Name:** larahub/api-response-kit

### Purpose

LaraHub API Response Kit is a Laravel package designed to standardize API responses, errors, and validation outputs across Laravel applications. The package aims to eliminate inconsistent API formats, reduce boilerplate code, and enforce a unified response structure across teams and projects.

### Core Objectives

* Provide a unified API response schema.
* Automatically format successful responses and errors.
* Standardize validation error responses.
* Reduce repetitive response code in controllers.
* Improve team-level consistency and API predictability.

---

## 2. Target Users

* Laravel backend developers
* API-driven applications (REST APIs)
* Development teams requiring standardized API contracts
* SaaS and enterprise Laravel projects

## 3. Key Features (v1 Scope)

### 3.8 Environment-Aware Debug Mode

* Automatically detects application environment (APP_ENV / APP_DEBUG).
* In debug mode (development), full error details and stack traces are included.
* In production mode, sensitive information (SQL queries, stack traces, file paths) is hidden.
* Ensures security and clean API responses in production.

### 3.9 Custom Response Format Support

* Allows developers to define their own response structure.
* Developers can create custom formatter classes.
* Package dynamically loads user-defined formatters via configuration.
* Enables multiple response schemas for different projects or teams.

### 3.1 Automatic API Response Formatting

* Automatically wraps controller return values into a standard JSON structure.
* Works with arrays, Eloquent models, and collections.

### 3.2 Global Error Handling

* Converts exceptions into standardized JSON error responses.
* Handles HTTP errors (404, 403, 401, 500, etc.).

### 3.3 Validation Error Standardization

* Formats Laravel validation errors into a clean and predictable structure.

### 3.4 Configurable Response Schema

* Allows developers to customize response keys via a config file.

### 3.5 Request ID Generation

* Generates a unique request ID for each API request.
* Useful for debugging and logging.

### 3.6 Facade Support

* Provides an optional facade for manual response control.

### 3.7 Middleware Integration

* Intercepts outgoing responses and applies formatting automatically.

## 4. Standard Response Schema (Default)

### 4.1 Success Response

```json
{
  "success": true,
  "message": "Success",
  "data": {},
  "meta": {
    "request_id": "LH-XXXXXX",
    "timestamp": "2026-01-27T12:00:00Z"
  }
}
```

### 4.2 Error Response

```json
{
  "success": false,
  "message": "Error message",
  "errors": null,
  "meta": {
    "request_id": "LH-XXXXXX",
    "timestamp": "2026-01-27T12:00:00Z"
  }
}
```

### 4.3 Validation Error Response

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": {
    "email": "The email field is required"
  },
  "meta": {
    "request_id": "LH-XXXXXX",
    "timestamp": "2026-01-27T12:00:00Z"
  }
}
```

## 6. Core Components

### 6.6 Environment Error Filter

Responsibilities:

* Detect application environment (development vs production).
* Filter sensitive error details such as SQL queries, stack traces, and internal paths.
* Provide safe error messages in production mode.

### 6.7 Custom Formatter Engine

Responsibilities:

* Allow developers to register custom response formatter classes.
* Dynamically resolve formatter based on configuration.
* Provide an interface for extending response formats.

### 6.1 Service Provider

Responsibilities:

* Register package services.
* Publish configuration file.
* Bind ApiResponseKit service into Laravel container.

### 6.2 Middleware (ApiResponseKitMiddleware)

Responsibilities:

* Intercept outgoing HTTP responses.
* Detect response type (success, error, validation error).
* Apply appropriate formatter.

### 6.3 Formatter Classes

#### SuccessFormatter

* Formats successful responses.

#### ErrorFormatter

* Formats generic errors and HTTP exceptions.

#### ValidationFormatter

* Formats validation error responses.

#### ExceptionFormatter

* Converts exceptions into standardized JSON errors.

### 6.4 Support Classes

#### ResponseSchema

* Builds final response structure based on config.

#### RequestIdGenerator

* Generates unique request IDs.

#### ConfigResolver

* Resolves configuration values dynamically.

### 6.5 Facade (ApiResponseKit)

* Provides manual control when automatic formatting is not desired.

## 7. Configuration

### 7.1 Debug Mode Configuration

```php
return [
    'debug' => [
        'show_trace' => env('APP_DEBUG', false),
        'hide_sql_errors' => env('APP_ENV') === 'production',
    ],
];
```

### 7.2 Custom Formatter Configuration

```php
return [
    'formatter' => LaraHub\ApiResponseKit\Formatters\DefaultFormatter::class,

    'custom_formatters' => [
        'my_custom_format' => App\Formatters\MyCustomFormatter::class,

];
```

## 13. Technical Architecture Diagram (v1)

### 13.1 High-Level Request–Response Flow

```
Client Request
      │
      ▼
Laravel HTTP Kernel
      │
      ▼
Global Middleware Stack
      │
      ▼
ApiResponseKitMiddleware (Core Interceptor)
      │
      ▼
Controller / Service / Repository Layer
      │
      ▼
Raw Laravel Response (Model / Array / JsonResponse / Exception)
      │
      ▼
ApiResponseKit Engine
      │
      ├── Environment Detector (Debug / Production)
      ├── Response Type Resolver
      │      ├── Success Response
      │      ├── Validation Error
      │      ├── HTTP Exception
      │      └── System Exception
      ├── Formatter Resolver
      │      ├── Default Formatter
      │      └── Custom User Formatter
      ├── Response Schema Builder
      ├── Request ID Generator
      └── Error Sanitizer (Production Mode)
      │
      ▼
Standardized API JSON Response
      │
      ▼
Client (Frontend / Mobile / Third-party)
```

---

### 13.2 Internal Engine Architecture (Core Components Interaction)

```
ApiResponseKitMiddleware
        │
        ▼
ResponseResolver
        │
        ├── detects response type
        ├── inspects exception or payload
        ▼
FormatterManager
        │
        ├── loads default formatter
        ├── loads custom formatter (if configured)
        ▼
ResponseFormatterInterface
        │
        ├── SuccessFormatter
        ├── ValidationFormatter
        ├── ErrorFormatter
        └── ExceptionFormatter
        ▼
ResponseSchema
        │
        ├── merges config keys
        ├── attaches meta data
        ▼
EnvironmentFilter
        │
        ├── debug mode: show full trace
        └── production mode: hide SQL & sensitive data
        ▼
Final JSON Response
```

---

### 13.3 Exception Handling Flow (Engineering Logic)

```
Exception Thrown
      │
      ▼
Laravel Exception Handler
      │
      ▼
ApiResponseKit Exception Listener
      │
      ├── classify exception type
      │      ├── ValidationException
      │      ├── HttpException
      │      ├── QueryException (SQL)
      │      └── Generic Exception
      │
      ▼
ExceptionFormatter
      │
      ├── debug mode → full stack trace
      └── production mode → sanitized message
      ▼
Standard Error Response
```

---

### 13.4 Custom Formatter Extension Flow

```
Developer Creates Custom Formatter
(App\Formatters\MyCustomFormatter)
        │
        ▼
Registers Formatter in Config
(api-response-kit.php)
        │
        ▼
FormatterManager Loads Custom Formatter
        │
        ▼
ResponseFormatterInterface Contract
        │
        ▼
Custom Response Schema Applied
        │
        ▼
Final API Response
```

### 13.5 Key Engineering Principles Used

* Middleware-based interception
* Strategy Pattern (Formatter system)
* Factory Pattern (Response type resolver)
* Dependency Injection (Service container)
* Open/Closed Principle (custom formatters)
* Environment-aware error handling
